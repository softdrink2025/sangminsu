# 🚀 12/5 관통프로젝트 10

# 📚 학습한 내용

- **문제 정의**
  - “왜 이 프로젝트가 필요한가?”를 바탕으로 문제 정의
  - 실제로 불편하거나 개선이 필요한 문제를 명확하게 정의

- **목표 설정**
  - 누구를 위한? 무엇을? 어떻게? 왜 필요한가? 이미 존재하는 서비스와의 차별점은?

- **아이디어 구체화 과정**
  - 서비스명, 서비스 개요, 주요 기능, 주요 페이지 구성 상세히 작성
  - 각각의 요구사항에 따라 요구사항 상세를 적고 얼마나 우선시 되는 지 기입

- **진행 계획 수립**
  - `WBS`와 `Gantt Chart`를 통해 정의된 요구사항을 기준으로 구체적인 일정을 계획하고 문서화 진행
  - `WBS`
    - 프로젝트의 전체 범위를 계층적이고 체계적인 방식으로 더 작고 관리하기 쉬운 구성 요소로 분해하는 도구
  - `Gantt Chart`
    - 프로젝트 전체 일정을 확인할 수 있는 바 형태의 도구
    - `WBS`에서 분해된 작업 패키지에 예상 기간, 선행 관계 등을 고려하여 작성

- **데이터 탐색**
  - 실제 서비스 제공을 위해서 필요한 데이터셋 찾기
  - 추천 시스템의 구현을 위한 목적성에 알맞은 데이터셋 수집

- **프로젝트 설계**
  - 구체적인 프로젝트의 기획이 완료되었다면 프로젝트 구조를 묘사하는 다이어그램을 작성한다

- **다이어그램 작성**
  - `ERD`
    - Entity Relationship Diagram
    - 개체-관계 모델, 개별 데이터의 속성을 정의하고, 개체별로 분류 후 각 개체 간의 관계를 명확히 표현한 다이어그램
  - `Use Case Diagram`
    - 사용자와 시스템 사이 가능한 상호 작용을 시각적으로 묘사한 다이어그램

- **UI/UX 설계**
  - 서비스 전체의 화면 구조를 설계하여 사용자 흐름을 정리
  - 화면에서 화면 간 이동 흐름을 묘사

- **API 명세 작성**
  - 필요한 화면을 기준으로 서버에서 제공해야 하는 엔드포인 구조를 설계
  - 각 기능별로 API를 분리하여 작성
  - Method, URL, 설명, 요청/응답, 인증 필요 여부를 포함

# 👨‍💻 어려웠던 부분

- `ERD`를 작성할 때 엔터티 간의 관계를 설정하는 것이 어려웠습니다.
- 사용자에게 편리한 UI/UX 설계하는 과정이 힘들었습니다.
- API로 데이터셋을 확보할 때, 필요한 데이터를 가공하고 추출하는 부분이 어려웠습니다.
- 아이디어를 구체화하고 생각하는 과정이 어려웠습니다.

# ✨ 새로운 배운 것

- 프로젝트를 단순히 구현하는 것 보다 정확하고 올바른 설계가 진행되어야 프로젝트 구현 과정이 줄어들고 쉽게할 수 있다고 알게 되었습니다.
- `ERD` 또는 `Use Case Diagram`를 만드는 방법과 그것을 도와주는 사이트가 있다는 사실을 알게 되었습니다.

# 💡 느낀점

- 프로젝트를 구현하는 것보다 프로젝트를 기획하고 설계하는 것이 어렵고 기획 단계에서 얼마만큼 구체적으로 생각 해야하는지 감이 잘 잡히지 않았습니다.
- 생각보다 기획 단계에서 진행해야 할 내용이 너무 많았고, 좋은 설계가 있어야 구현할 때 힘들지 않다는 것을 느꼈습니다.
